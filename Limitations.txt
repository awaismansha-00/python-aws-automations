Below are the main limitations and caveats for every script.

# 1. AWS Resource Inventory

* Scans only the region defined in `REGION`.
* Pagination is required for large accounts; missing it may return incomplete results.
* Listing a resource does not necessarily show its full configuration.
* Some resource details require additional API calls.
* API permissions may differ between services even if the script uses one AWS identity.

# 2. Unused Resource Detector

### Unattached EBS volumes

* An `available` volume is unattached, but it may still be intentionally retained.
* Recently detached volumes may be waiting for reuse.
* Tags such as `Keep=true` should be checked before recommending deletion.
* Storage cost estimates require current regional pricing; Boto3 does not automatically calculate exact cost.

### Elastic IPs

* An unassociated Elastic IP may be reserved intentionally.
* Some associations may exist through network interfaces rather than directly through an EC2 instance.

### Stopped EC2 instances

* A stopped instance can still incur costs for EBS volumes, Elastic IPs and snapshots.
* `LaunchTime` does not show when the instance was stopped.
* Determining how long it has been stopped requires CloudTrail, AWS Config or another tracking method.
* Some stopped instances may be intentionally preserved.

### Old snapshots

* An old snapshot is not automatically unnecessary.
* Snapshots can be required for compliance, disaster recovery or long-term retention.
* A snapshot may be used by an AMI.
* Incremental snapshot storage means snapshot size does not directly equal billed storage.

# 3. EC2 Start/Stop Scheduler

* Only instances with `AutoSchedule=true` are selected.
* The script currently uses a manual `ACTION` variable rather than actual time-based scheduling.
* It must be run by EventBridge, Lambda, cron or another scheduler for true automation.
* Stopping an EC2 instance does not remove EBS storage charges.
* Instance-store data may be lost when stopping supported instances.
* Some instance types or configurations cannot be stopped.
* Auto Scaling Groups may recreate or replace stopped instances.
* Production instances should have exclusion tags such as `DoNotStop=true`
* Start and stop operations are asynchronous; the script does not currently wait until the final state is reached.

# 4. EBS Snapshot Manager

* Volumes must have `Backup=true` to be selected.
* Creating a snapshot while an application is writing data may produce a crash-consistent rather than application-consistent backup.
* Databases may require filesystem freezing or application-aware backup procedures.
* Snapshot creation is asynchronous.
* The script reports the initial state but does not wait for completion.
* Repeated runs may create duplicate snapshots.
* The current version does not check whether a recent snapshot already exists.
* Snapshots used by AMIs should not be deleted.
* Snapshot tags should include environment, owner and retention policy in production.

# 5. AWS Cost Report

* AWS Cost Explorer must be enabled.
* Cost Explorer API calls may themselves have a small charge.
* Billing data is not real-time and can be delayed.
* The end date is exclusive.
* Current-day costs can still change after the script runs.
* Unblended cost is only one way of measuring cost.
* Savings Plans, Reserved Instances, credits, refunds and taxes may require different metrics.
* Grouping by service may hide resource-level causes.

# 6. Tag Compliance Checker

* The Resource Groups Tagging API does not return every AWS resource type.
* Completely untagged resources may not always appear through this API.
* The script checks whether tag keys exist, but does not validate values.
* A tag such as `Environment=""` may technically exist but still be invalid.
* Tag key capitalisation matters.
* Organisation-wide policies may be better enforced with AWS Organizations Tag Policies or AWS Config.

# 7. Security Group Auditor

* Public access is not automatically a vulnerability; ports 80 and 443 are commonly public.
* Severity depends on the workload and network architecture.
* Rules using prefix lists or referenced security groups require separate interpretation.
* It does not check whether a security group is actually attached to a resource.
* A public security-group rule may still be protected by a Network ACL, firewall or application authentication.
* IPv6 exposure must be checked separately through `::/0`.
* Port ranges can expose sensitive ports even if the exact port is not written as a standalone rule.

# 8. IAM Security Auditor

* Missing MFA is most relevant for users with console access.
* Directly attached policies are not always insecure; they are mainly harder to manage.
* Old access keys may still be required by production systems.
* “Never used” information can be incomplete or delayed.
* Access-key last-used data may not update immediately.
* Wildcard permissions may sometimes be required, although they should be justified.
* Automated IAM remediation is risky and should not be enabled without review.

# 9. CloudWatch Alarm Auditor

* It currently checks only EC2 instances.
* It checks only for `CPUUtilization` and `StatusCheckFailed`.
* The existence of an alarm does not mean the alarm is correctly configured.
* The script does not validate:

  * threshold
  * period
  * evaluation periods
  * missing-data behaviour
  * SNS actions
  * alarm-action status
* Metric-math alarms may not be identified correctly through simple `MetricName` matching.
* Composite alarms are not included.
* Alarm dimensions must contain `InstanceId` for the matching logic to work.
* Stopped instances may legitimately have no useful CPU alarm.

# 10. CloudWatch Operational Health Report

* It currently reports only running EC2 instances.
* CPU alone is not enough to determine application health.
* High CPU may be expected for batch workloads.
* Low CPU does not guarantee that an application is healthy.
* `StatusCheckFailed` only indicates EC2 instance or system-level issues.
* Basic monitoring uses five-minute intervals unless detailed monitoring is enabled.
* The script performs multiple API calls per instance, which is inefficient at scale.
* `get_metric_data()` would be more efficient for many resources.
* Memory and disk usage are unavailable by default unless the CloudWatch Agent publishes them.

# 11. Systems Manager Remote Health Checker

* Instances must be registered as Systems Manager managed nodes.
* SSM Agent must be installed and running.
* Each instance needs the correct IAM instance profile.
* Instances need connectivity to SSM endpoints through internet access or VPC endpoints.
* The script supports Linux only because it uses `AWS-RunShellScript`.
* Windows requires `AWS-RunPowerShellScript`.
* Command execution is asynchronous.
* Run Command uses eventual consistency, so invocation results may not appear immediately.
* Commands can time out.
* Large output may be truncated.
* Running operational commands across many instances can create load.
* Disruptive operations should require confirmation and concurrency controls.

# 12. ECR Image Cleanup

* Old images may still be used by ECS, EKS, Lambda or rollback procedures.
* An untagged image may still be referenced by digest.
* Protected tags such as `prod` and `latest` do not guarantee that all active images are protected.
* The current “keep latest images” logic counts all images in the repository together.
* It does not retain images separately by application branch or tag pattern.
* Image size reported by ECR may not exactly match total physical storage savings.
* Native ECR lifecycle policies are usually better for continuous cleanup.
* Replicated images and cross-region repositories require additional consideration.
* Deletion batches have API limits.
* Shared image digests can have multiple tags.

# 13. Lambda Rollback Helper

* Rollback works only when published Lambda versions already exist.
* `$LATEST` cannot be used as a stable immutable rollback target.
* The function must have an alias such as `prod`.
* The script only changes the alias; it does not restore environment variables or external infrastructure.
* A previous Lambda version may be incompatible with:

  * current database schema
  * updated event formats
  * new IAM permissions
  * changed external services
* It does not support weighted alias routing in the basic version.
* Updating the alias changes traffic quickly and can affect production immediately.
* Rollback does not undo side effects already created by the faulty version.

# 14. Route 53 and ACM Auditor

### Route 53

* Listing DNS records does not confirm that targets are healthy.
* Alias records may point to services that require service-specific verification.
* The current version does not detect stale ALB, CloudFront or API Gateway targets.
* Private hosted zones may contain records that are intentionally inaccessible publicly.
* DNS record values may be split across multiple entries.

### ACM

* CloudFront certificates must normally be checked in `us-east-1`.
* An unattached certificate may be intentionally reserved.
* ACM-managed certificates usually renew automatically when validation remains valid.
* Expiry warnings do not necessarily mean renewal will fail.
* Imported certificates require different renewal handling.
* The script does not verify DNS validation records.

# 15. S3 Governance Auditor

* `list_buckets()` returns buckets across the whole account.
* Bucket configuration APIs may need to be called through the correct regional endpoint.
* Bucket-level Block Public Access is not the whole picture; account-level settings also apply.
* A disabled bucket-level setting does not automatically mean the bucket is public.
* ACLs and bucket policies must also be inspected for actual public access.
* Modern S3 automatically encrypts new objects with SSE-S3, so missing explicit encryption configuration does not necessarily mean data is unencrypted.
* Versioning is not required for every temporary or disposable bucket.
* Access logging is not always required and can create extra storage cost.
* Lifecycle policies depend on retention requirements.
* Logging buckets must be configured carefully to avoid recursive logging.

# 16. Multi-Account AWS Inventory

* Requires AWS Organizations.
* Must run from the management account or an authorised delegated administrator.
* The target role must exist in every member account.
* The current identity must have permission to call `sts:AssumeRole`.
* Role trust policies must allow the management identity.
* Suspended or inaccessible accounts are skipped.
* It scans only one configured region.
* Large organisations may hit API throttling.
* Different accounts may use different role names or permission models.
* Service Control Policies may block operations even after role assumption.
* The script currently inventories only EC2 instances.
* It does not scan resources in the management account separately unless the role also exists there.
* External IDs may be required in some cross-account configurations.
* Errors in one account should not stop the entire scan.
* The older Organizations `Status` field is being replaced by `State`, so code should handle the newer field.

# General caveats for all scripts

* Avoid hard-coded credentials.
* Use IAM roles or the AWS credential chain.
* Use least-privilege permissions rather than administrator access in production.
* Use paginators for list and describe operations.
* Expect API throttling and add retries where appropriate.
* Catch `ClientError` and log the AWS error code and message.
* Treat missing response fields safely with `.get()`.
* Keep destructive actions disabled by default.
* Use dry-run, confirmation and exclusion tags before modifying resources.
* Test in a non-production account first.
* Record all changes in logs.
* Add unit tests using mocks before using scripts operationally.
* Scan all required regions and accounts.
* Never assume that “old”, “unused” or “unattached” automatically means “safe to delete”.
* AWS resource relationships should be checked before remediation.
* For recurring production automation, Lambda, EventBridge, AWS Config, Systems Manager Automation or native lifecycle policies may be more reliable than running scripts manually.
